# Requirements: DVConf

**Defined:** 2026-03-04
**Core Value:** Users can hold real-time video conferences through decentralized relay nodes where honest operators are economically rewarded and dishonest ones are slashed — all verifiable on-chain.

## v1 Requirements

Requirements for Milestone 1: Foundation + Registries + Off-chain Daemons.

### Foundation (Phase 1 Re-review)

- [ ] **FOUND-01**: DVCONF fungible token deployed on Sui with treasury cap
- [ ] **FOUND-02**: NetworkRegistry singleton stores global config (scoring weights, reward ratios, thresholds)
- [ ] **FOUND-03**: Scoring weights sum to 10,000 basis points; reward ratios sum to 10,000
- [ ] **FOUND-04**: Node registration requires minimum stake (aborts E_INSUFFICIENT_STAKE)
- [ ] **FOUND-05**: Registered nodes can top up stake and withdraw (when unlocked)
- [ ] **FOUND-06**: Unregister aborts with E_STAKE_LOCKED if locked, E_NOT_OWNER if not owner
- [ ] **FOUND-07**: All cap constructors are public(package) — no external cap minting
- [ ] **FOUND-08**: Registration and top_up_stake abort E_PROTOCOL_PAUSED when paused
- [ ] **FOUND-09**: MinerStore tracks validator set with package-private access
- [ ] **FOUND-10**: CP queries module provides external read access to validator set
- [ ] **FOUND-11**: All error codes match namespace table (100s network_registry, 200s staking, 300s miner_store, 400s registration)
- [ ] **FOUND-12**: All 34 existing tests pass with no regressions

### Registry Layer (Phase 2)

- [ ] **REG-01**: RoomManager singleton can create room entries and track active room count
- [ ] **REG-02**: RoomManager stores per-room rules (min_relay, min_cp, min_validator)
- [ ] **REG-03**: ControlPlaneRegistry allows CP node registration with stake requirement
- [ ] **REG-04**: CP nodes submit heartbeat() to prove liveness; missed heartbeats mark inactive
- [ ] **REG-05**: ControlPlaneRegistry tracks available CPs and room assignments
- [ ] **REG-06**: RelayRegistry allows relay node registration with declared mode (SFU or MCU)
- [ ] **REG-07**: RelayRegistry stores validator_probed_rtt per relay (ground-truth, not self-reported)
- [ ] **REG-08**: RelayRegistry tracks current load per relay (update_load)
- [ ] **REG-09**: ValidatorRegistry allows validator registration with session wallet mapping (dual-key)
- [ ] **REG-10**: ValidatorRegistry maps session addresses to validator IDs
- [ ] **REG-11**: UserRegistry allows user profile registration
- [ ] **REG-12**: All registries are independent shared objects (no cross-registry contention)
- [ ] **REG-13**: Error code namespaces pre-assigned for all Phase 2 modules (500-1099 range)
- [ ] **REG-14**: All registry entry points check paused flag on state-mutating operations
- [ ] **REG-15**: All registries emit events for off-chain daemon consumption

### Off-chain Daemons (Phase 5 partial)

- [x] **DAEMON-01**: Signaling node exchanges WebRTC ICE candidates via WebSocket
- [x] **DAEMON-02**: Signaling node is stateless and does not depend on chain state
- [x] **DAEMON-03**: Control Plane daemon subscribes to Sui events (room creation, relay updates)
- [x] **DAEMON-04**: CP daemon runs relay scoring algorithm using on-chain data (reputation, RTT, load, stake, region)
- [x] **DAEMON-05**: CP daemon submits relay assignment votes on-chain (scoped: logs scores, votes deferred to v2)
- [x] **DAEMON-06**: CP daemon sends heartbeat() to ControlPlaneRegistry at configured interval
- [x] **DAEMON-07**: CP daemon uses exponential backoff on chain interaction failures
- [x] **DAEMON-08**: Validator daemon joins rooms disguised as regular user (session wallet)
- [x] **DAEMON-09**: Validator daemon measures packet integrity, latency, loss, bytes forwarded
- [x] **DAEMON-10**: Validator daemon submits dual-key signed SessionProof on-chain
- [x] **DAEMON-11**: All daemons share types via monorepo shared package
- [x] **DAEMON-12**: All daemons use @mysten/sui SDK for chain interactions

## v2 Requirements

Deferred to future milestones.

### Room Lifecycle (Phase 3)

- **ROOM-01**: Room state machine (Pending -> Ready -> Active -> Closed)
- **ROOM-02**: CP relay assignment voting with >= 2/3 consensus
- **ROOM-03**: Validator secret assignment (commit-reveal or off-chain assignment)
- **ROOM-04**: Join/leave room entry points
- **ROOM-05**: Event emission for off-chain state changes

### Economic Layer (Phase 4)

- **ECON-01**: Escrow tokens on room creation
- **ECON-02**: SessionProof structure with work + quality metrics + dual-key fields
- **ECON-03**: Dual-key proof submission + verification
- **ECON-04**: Median metric aggregation from multiple validators
- **ECON-05**: Quality multiplier calculation (packet loss thresholds)
- **ECON-06**: Work-based relay reward distribution
- **ECON-07**: Validator accuracy scoring + reward
- **ECON-08**: Slashing logic
- **ECON-09**: RTT writeback to RelayRegistry

### Relay Nodes (Phase 5 remainder)

- **RELAY-01**: SFU relay node (mediasoup stream fan-out)
- **RELAY-02**: MCU relay node (mediasoup/GStreamer stream mixing)

### Client App (Phase 6)

- **CLIENT-01**: Registration flow (wallet-based)
- **CLIENT-02**: Create room flow
- **CLIENT-03**: Join room + WebRTC connect
- **CLIENT-04**: SFU session view
- **CLIENT-05**: MCU session view

## Out of Scope

| Feature | Reason |
|---------|--------|
| CDN integration | Infrastructure complexity without proving thesis |
| Recording/storage | Requires CDN, storage billing, privacy compliance |
| Chat/messaging | Separate product domain, thesis scope creep |
| Mobile native app | Web-first; mobile browsers work for demo |
| ZK proofs for validator identity | Thesis-sized project on its own; dual-key sufficient |
| E2E encryption | Incompatible with MCU mode; DTLS-SRTP already encrypts |
| Virtual backgrounds/filters | GPU/WASM complexity, no thesis value |
| Governance mechanism for BASE_RATE | Use fixed value; governance is separate research |
| Self-reported relay metrics | Explicitly eliminated in spec rev 4 |
| Screen sharing | Defer until core video works |
